# 模子の衣柜 — 踩坑经验

> 项目至今所有的坑的汇总。每条结构：**症状 → 根因 → 解法 → 教训**。
> 标 **【待补充】** 的地方等 Victor 补细节；新坑直接往对应小节或末尾「Victor 补充」加。

## 一、Firebase / Firestore

### 0. 项目能跑起来前，Firebase 要先配一堆（最卡人的前置）
- **准备清单（按踩坑顺序）**：
  - 建 Firebase 项目 + 开 **Firestore**。注意 AI Studio / Firebase Studio 生成的是**命名数据库 `ai-studio-...`，不是 `(default)`**（见坑 3）。
  - 开 **Authentication → Google 登录**（本项目只用 Google），并把 **localhost + Vercel 域名**加进授权域名 / OAuth 同意屏幕。
  - 拿 Firebase 配置（apiKey/projectId/appId/authDomain/**firestoreDatabaseId**/storageBucket…）→ 放 `firebase-applet-config.json`，**必须带命名库 ID**，否则连的是 default 库。
  - 写 `firestore.rules` 并部署（只验 auth+userId、不验字段，见坑 4；部署认命名库，见坑 2/3）。
  - 配环境变量：`KIMI_API_KEY`(服务端) + `VITE_AUTHOR_UID`(构建期，见坑 10)——本地 `.env` 与 Vercel 各一份。
- **教训**：「项目跑不起来」八成卡在这些前置（库选错 / 规则没部署 / 授权域名没加 / env 没同步），不是代码问题。
- 【待补充：你配置时卡过的具体步骤、控制台里哪步最反直觉、OAuth 同意屏幕 / 授权域名的细节】

### 0.5〔惨痛〕一次账号迁移把数据库整个删没了
- **症状**：某次数据迁移，直接把数据库整个删没了。
- **根因**：账号迁移时**比对/检测了错误的 ID**（拿错 uid 去匹配），把数据迁/清到了错的地方；**且原账号数据没有备份**，错了就没法回退。
- **解法 / 恢复**：没有备份 → **只能重建**（数据丢了，重新录入 / 导入）。
- **教训**：任何迁移 / 批量 / 覆盖操作前，**先导出备份**（Firestore export / 控制台 / 脚本 dump）；破坏性操作先在**测试库**演练，别直接对生产库跑。**尤其涉及 uid / 账号匹配的迁移，先 dry-run 把 ID 核对清楚再真正写。**

### 1. Storage 上传一直 permission-denied
- **症状**：图片上传到 Firebase Storage 总是 permission-denied。
- **根因**：新格式 Storage bucket 默认规则 `allow write: if false`，开箱不可写。
- **解法**：**弃用 Storage，图片压缩成 base64 直接存 Firestore 文档字段**（720px / 78% JPEG，`src/lib/cropImage.ts` 的 `compressToBase64()`）。
- **教训**：图床这条路在这个项目里直接放弃了；但 base64 存库埋下了后面「读取额度 + 带宽」两个大坑（见第三节）。

### 2. 改了 Firestore 规则不生效 / 部署 "skipping upload"
- **症状**：`firebase deploy` 后规则没变。
- **根因**：`firebase.json` 里 rules 路径写错（用了冒号不是点号）。
- **解法**：`"rules": "firestore.rules"`；部署后确认输出是 **"uploading rules"** 而非 "skipping upload"。

### 3. 命名数据库（最隐蔽）
- **症状**：在 Firebase 控制台改了规则，线上还是旧规则。
- **根因**：这个项目用的是 **命名数据库 `ai-studio-6fd5f2f5-...`，不是 `(default)`**。控制台默认显示 default 库的规则。
- **解法**：改规则前先在控制台顶部**切换到命名库**；CLI 部署也认 `firebase.json` 里配置的库。
- **教训**：AI Studio / Firebase Studio 生成的项目常是命名库，且是「免费层数据库」，后面额度坑就和它有关。

### 4. 规则加字段校验就 permission-denied
- **症状**：给规则加了字段类型/长度校验后，写入全被拒。
- **根因**：base64 图片字段超长，触发校验失败。
- **解法**：规则**只校验 auth + userId，不做字段类型校验**。

### 5. 手机登录每次 UID 都不一样
- **症状**：手机上每次登录拿到的 UID 不同，数据对不上。
- **根因**：`signInAnonymously` 和 `signInWithRedirect` 竞争。
- **解法**：**移除匿名登录**，只保留 Google 登录，彻底解决。

### 6.〔本次最大坑〕「未开启分享」其实是读取额度被刷爆
- **症状**：公开衣柜页报「未开启分享」，但分享开关明明是开着的。
- **根因链**：Firestore 免费层**每日读取额度（5 万/天，此命名库硬上限，开账单也提不了）耗尽 → 返回 429** → 安全规则 `sharingEnabled()` 需要 `get(wardrobe_users/{uid})` 验证开关，这条读被额度挡掉 → 规则判 false → 前端误显示「未开启分享」。
- **解法**：① 额度按**太平洋时间午夜**重置（≈北京次日下午）；② 代码区分 `permission-denied`（真没开）vs `resource-exhausted/429`（显示「服务器繁忙稍后再试」，且**不缓存错误**）；③ 根治见第三节「边缘缓存」。
- **教训**：**看到「权限/分享坏了」，先查底层 HTTP 状态是 429（额度）还是 403（真没权限），别信 UI 文案。** 一个限流能伪装成权限 bug。

### 7. 真凶是「每次访问读全量」，不是实时监听
- **症状**：读取次数异常高，怀疑 onSnapshot 监听泄漏。
- **根因**：用量图里**实时读取只有 ~200，写入只有 ~480**，而总读取 3~6 万——说明监听不是元凶，**公开页每次打开都 `getDocs` 读「全部 ~101 件单品」**才是。Firestore **按「文档数」计费，不是字节**：一次查询返回 101 条 = 101 次读。
- **教训**：排查额度先看「实时读取 vs 一次性读取」的占比，能快速定位是监听还是查询的问题。

### 8. 调试本身把额度刷爆了
- **症状**：「我就打开两次怎么就 3 万读取？」
- **根因**：单次公开页打开 ≈ 145 读（101 单品 + 43 搭配 + 规则 1）。**开发期反复加载重页面 + 脚本直连 Firestore 探测**，几十次就上万。用量曲线呈「工作时段离散尖峰 + 凌晨贴零」= 典型开发自损，不是真实流量。
- **教训**：**调试时不要 hammer live 的重页面**；验证优先走缓存接口、批量化、能本地就别打线上。

## 二、Vercel / Serverless 函数

### 9.〔ESM 坑〕serverless 函数裸 import JSON → FUNCTION_INVOCATION_FAILED
- **症状**：新加的 `api/public/[uid].ts` 一调用就 500 `FUNCTION_INVOCATION_FAILED`，日志 `TypeError [ERR_...]`。
- **根因**：`package.json` 是 `"type": "module"`，函数按 **ESM** 运行；ESM 下 `import cfg from './x.json'` 需要 import assertion，否则运行时抛错（构建却能过，坑在运行时）。
- **解法**：**别在函数里 import JSON**；把需要的非密钥常量（projectId、dbId）**硬编码**进函数（它们本就在前端 bundle 里公开）。
- **教训**：前端能 `import json`（Vite 打包）≠ serverless 能（Node ESM 运行时）。函数里少依赖、值能内联就内联。

### 10. VITE_ 变量是构建期注入，本地和 Vercel 是两套
- **症状**：改了 `VITE_AUTHOR_UID`，本地好了线上还是旧的。
- **根因**：`VITE_` 前缀变量**打包时写死**（构建期），不是运行时读。本地 `.env` 和 Vercel 环境变量是**两套独立**配置。
- **解法**：改 `VITE_*` 要**两处都改 + 重新构建/部署**（本地改 `.env` 重启 dev；线上改 Vercel env 后 Redeploy/push）。
- **教训**：`.env` 通常 gitignore，所以线上完全靠 Vercel 后台那份；别以为 push 代码就带过去了。

### 11. Vercel Preview 部署有访问保护，沙箱/无登录态打不开
- **症状**：`vercel deploy` 出的 preview URL 浏览器打开一直超时。
- **根因**：Preview 默认开 **Deployment Protection**，需要 Vercel 登录态；无登录的浏览器/CI 会被挡（表现为超时/跳认证）。
- **解法**：测 serverless 要么用**生产公开域名**（push 后），要么配 Protection Bypass token。
- **教训**：「preview 打不开」不一定是部署失败——先确认是不是被保护挡了。

### 12. `vercel env add` 给 Preview 环境的 CLI 怪癖
- **症状**：给 Production/Development 加变量都成功，唯独 Preview 一直报 `git_branch_required`，加 `main` 又说 main 是 production branch。
- **根因**：Preview 变量要绑 git 分支；「所有 preview 分支」那条命令在这个 CLI 版本有 bug。
- **解法**：Preview 变量非必须就**先跳过**（只影响 PR 预览，不影响线上），要配就去网页后台加。

### 13. `npm run dev`(vite) 不 serve /api 函数
- **症状**：本地前端能跑，但 `fetch('/api/...')` 404。
- **根因**：`npm run dev` 只起 Vite；`/api` 的 serverless 函数要 Vercel 运行时。
- **解法**：本地测函数用 **`vercel dev`**，或直接推 preview/prod 验证。

### 14. 边缘缓存：怎么判断到底缓没缓
- **要点**：响应头返回给浏览器的 `Cache-Control` 可能被 Vercel 转换成只剩 `public`（不显示 s-maxage），**别以此判断**。看 **`x-vercel-cache: HIT/MISS` + `age`** 才准。
- **要点**：**每次部署都会清空边缘缓存** → 部署后第一批请求必然 MISS 回源。

### 14.5〔重要〕「调试期读取依然 1 万」的真相 = 部署清缓存（deploy churn）
- **症状**：防限流（边缘缓存）都上了，调试时一次打开还是涨上万读取，像是没生效。
- **诊断（实测）**：稳态下（缓存焐热、近期没部署）打开公开页 —— metadata 和图片**全部 `x-vercel-cache: HIT`，0 次 Firestore 读**；但函数日志里短窗口有 **~98 次 `/api/img` 冷调用**。
- **根因**：**每次部署清空整个边缘缓存**。Phase 2 把图片拆成 ~101 个独立缓存对象后，**一次部署 = 作废 metadata + 全部 ~101 张图缓存** → 部署后首批访问把它们**全部回源重读**（~145 metadata + 最多 ~101 图）。开发期**频繁部署 + 反复刷新**，缓存永远焐不热 → 读取持续高。
- **结论**：**这是「部署抖动 + 调试自损」，不是生产问题**。真实访客（没人在部署时）命中热缓存，~0 读。
- **干净验证**：**停止部署**，打开页面焐热，再连刷 10 次 → 全 HIT、读取曲线基本不动。**别在刚部署完就量读取**。
- **教训**：Phase 2 之后「部署的缓存代价」更高了（多了 100+ 图片对象作废）；密集迭代时别拿「读取次数」当即时反馈，它会被部署污染。

【待补充：Vercel 带宽/用量告警、域名、构建超时等你遇到的坑】

## 三、架构决策（埋雷 & 拆雷）

### 15. base64 存 Firestore 的双重代价
- **背景**：当初为绕开 Storage 不可写，把图片 base64 存库（坑 1）。
- **代价**：① 单文档巨大；② Firestore **按文档数计费**，整柜查询 = 上百次读；③ 整柜 JSON 巨大（实测 **8.58 MB**）→ 同时压在**读取额度**和**带宽**两条线上。
- **教训**：一个「省事」的早期决策，会在「公开分享 + 流量」阶段连环爆雷。base64 存库适合自用，**不适合高频公开读**。

### 16.〔防限流正解〕公开页走边缘缓存接口（Phase 1）
- **做法**：公开页不直连 Firestore，统一走 `api/public/[uid].ts`（Firestore REST 读 + 校验 shareEnabled + 序列化 + `Cache-Control: s-maxage=3600, SWR`）。客户端用 `src/lib/publicWardrobe.ts` 的 `fetchPublicWardrobe()`。
- **效果**：Firestore **每作者每天只被读约 24 次，与访问量无关**；公开访问全部由 Vercel CDN 命中。
- **关键分离**：**owner 自己的 app 实时直连**（`WardrobeContext`/`BestMatchContext` 的 onSnapshot 不变），**只有公开读走缓存**。
- **配套坑**：缓存返回 `createdAt` 是 millis，owner 路径是 Firestore Timestamp → 统一用 `toDateSafe()` 兼容。

### 17.〔带宽优化〕图片拆分 + 懒加载（Phase 2）
- **问题**：Phase 1 解决了读取，但整柜 JSON 仍 8.58 MB（含 base64）→ 每访客都下 8.58 MB，吃 Vercel 免费 100GB/月。最离谱的是**单品深链也下整柜只为看一件**。
- **做法**：`api/public` **不再内联 base64**，把 `imageUrl`/`photoBase64` 改写成图片接口 URL `api/img/[uid]/[id].ts`（读单条解码成 JPEG，缓存 1 天，规则 gate）；卡片 `<img>` 公开路径切 `loading="lazy"`（`WardrobeItemCard` 的 `eager` prop）。
- **效果（实测）**：整柜 JSON **8.58MB → 92KB**；完整页首屏只加载视口内 ~15 张图；单品深链 ~160KB；免费带宽可撑量从 ~1.2 万 → ~10 万+ 次/月。
- **巧思**：让接口把 base64 换成 URL，**卡片组件 `<img src>` 几乎不用改**（它分不清 src 是 base64 还是 URL）。

### 18. 读取额度 ≠ 带宽（两个独立问题）
- **教训**：Phase 1 治读取、Phase 2 治带宽，**互不替代**。免费层要扛公开流量，两道都得做。

## 四、调试方法论 / 工具环境

### 19. 沙箱 shell 访问外部 HTTPS 不干净
- **现象**：bash `curl` 走代理对外 HTTPS 经常 `HTTP 000`/连接关闭；PowerShell 5.1 默认 TLS1.0 也连不上；Windows 下没有 `/tmp`。
- **解法**：验证线上行为优先用**浏览器（chrome-devtools）**跑 `fetch`/截图，比命令行可靠。

### 20. 用「用量曲线的形状」判断流量性质
- **教训**：读取暴涨时，看曲线是「离散尖峰 + 工作时段聚集」（=开发自损）还是「均匀分布」（=真实流量），能快速定性，少瞎猜。

### 21. 误报修复要分类错误码
- **教训**：前端遇到读失败，**区分 `permission-denied`（业务态：真没开/无权）和其它（系统态：额度/网络/繁忙）**，给不同文案，避免一个限流把用户/自己引到错误方向（见坑 6）。

## 五、AI 导入（Kimi 解析）

### 22.〔排错方向错了很久〕Kimi 一键导入总解析失败
- **症状**：用 Kimi 解析 PDF/文档一键导入，总是解析失败、各种报错。
- **走过的弯路**：一开始以为是**输出 JSON 不干净**，于是加了**强约束**——prompt 里严格要求只输出合法 JSON、禁 Markdown、末尾禁多余逗号，客户端再用正则提取 + 兜底闭合。这能修一部分，但没根治。
- **真根因**：**PDF 提取的文字量超过了 Kimi 单次可处理的容量**（输入/输出超限），返回被截断/报错。
- **解法**：**在代码里扩大单次可读取的最大值**（提高 `max_tokens` / 放宽一次性读取上限），让一次能吃下整篇。
- **教训**：解析类失败别只盯「输出格式」，**先确认是不是「输入/输出超长被截断」**——格式约束治标、容量治本，**两者其实都要**（约束保证干净，扩容保证不截断）。

## 六、一句话清单（速记版）
- 起步先配好 Firebase（命名库 / Google 授权域名 / 规则部署 / env 同步），跑不起来多半卡这。
- **迁移 / 批量 / 覆盖前先导出备份**，破坏性操作先在测试库演练（血泪，见坑 0.5）。
- 图床走不通就 base64，但别拿它做高频公开读。
- 改 Firestore 规则先确认在**命名库**上改。
- 「权限坏了」先看是不是 **429 额度**。
- 免费层按**文档数**计费 + **太平洋时间**重置 + 此库**硬上限不可提**。
- **公开读走边缘缓存**，owner 读保持实时；两者分离。
- serverless 函数（ESM）**别 import JSON**，常量内联。
- `VITE_*` 是**构建期**变量，本地与 Vercel **两处同步 + 重新部署**。
- 判断缓存看 **`x-vercel-cache`**；**部署会清缓存**，**别刚部署完就量读取**（deploy churn 会污染，见坑 14.5）。
- **调试别 hammer 重页面**，会真的花额度/带宽。
- AI 导入解析失败先查**是不是输入/输出超长被截断**（提容量），别只改输出格式约束。
