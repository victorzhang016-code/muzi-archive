# 衣 LOG 审美理解基座：分析规则

> 状态：生效中（本地实验层）  
> 引擎：`aesthetic-understanding-v1`  
> 适用范围：审美分析、搭配推荐、实验模式与未来的主题洞察。Production 数据与读图原始记录不在本规则下被改写。

## 1. 目的与边界

审美理解的任务，是从 Victor 已拥有的单品、评分、故事、Best Match、变体与视觉字段中，建立**有来源、可反驳、可校准**的个人穿搭理解。它不能把词频、图片模型输出或通用穿搭常识伪装成 Victor 的审美事实。

第一版是本地、只读、确定性的分析层。输入为本地衣橱快照；输出为独立的 `AestheticAnalysisBundle`：

```ts
{
  inputSnapshotHash,
  engineVersion,
  bodyProfile,
  evidence,
  relationClaims,
  metrics,
  insights,
  generatedAt,
}
```

引擎永远不得写回 `visionAnalyses`、单品、Best Match 或 Production 数据库。Victor 可以在本地审计台继续编辑 `proposed`，并通过“确认字段”将当前草稿升级为 `confirmed`；重新解析是显式动作，不能由批量任务覆盖正在校对的草稿。分析包可重算、导出与删除。

## 2. 输入与证据

| 证据                                             |     权重 | 可单独产生正式结论            |
| ------------------------------------------------ | -------: | ----------------------------- |
| `confirmed` 视觉字段、评分、Best Match 槽位/变体 |      1.0 | 可以                          |
| 单品故事、Best Match 原文                        |      0.9 | 仅限原文明确支持的表达        |
| `proposed` 视觉或文本字段                        |     0.55 | 不可以                        |
| 派生关系                                         | 继承来源 | 需要 confirmed 或直接原文支撑 |
| 无来源模型猜测                                   |        0 | 仅可显示为待处理，不进入结论  |

`proposed` 是积极探索的材料：它可参与聚类、候选关系与待验证假设。它不能单独构成“核心审美”“稳定原则”“身形效果”等正式判断。

每个证据至少保留：单品或 Best Match ID、字段、状态、权重、原文引用（如适用）与规则版本。

## 3. 评分、频率与文本的解释

- 单品评分是 Victor 的**综合偏好权重**，与穿搭兼容性分开计算，绝不合并成黑箱总分。
- 高评分 + 高搭配支持：审美锚点候选。
- 高评分 + 低搭配支持：情感锚点、场景专才或待开发机会。
- 低评分 + 高搭配支持：功能稳定器候选，不能因此被称为审美核心。
- Best Match 出现频率只代表档案复用；它不代表真实穿着频率、偏爱程度或重要性。
- “喜欢”“重要”“故事”等原文只能说明原句所指的对象和语义，不能被拆成无上下文的高频标签。

## 4. 关系语义

所有关系必须保留 `sourceMatchId`、槽位、主/变体身份、证据 ID、状态、权重与规则版本。

- **同套共现**：只表示某一套 Best Match 内的共同出现。
- **槽位变体**：仅在原 Best Match 的槽位上下文中表示可替换。
- **颜色呼应、构成支持、风格桥接**：必须能够回到具体字段与具体搭配。
- **情感/身份关系**：必须回到单品故事或搭配原文。
- **身形效果**：只来自 Victor 主动填写的比例感知、修饰目标或人工确认；系统禁止由图片推断身体条件。

关系图显示关系的类型和证据，不显示一个脱离语境的“兼容度”。

## 5. 单品角色

角色是候选，不是固定标签。一个单品可以同时拥有多个角色。

- 审美锚点：视觉签名、评分、故事或跨场景搭配支撑。
- 稳定器：负责承接、收束、降低冲突、保障可穿性。
- 桥梁：连接两个颜色、风格或正式度簇。
- 点缀：承担局部颜色、材质、细节或身份表达。
- 场景专才：在明确场合、季节、功能或身形目标下高价值。
- 情感锚点：由故事和身份价值支持，即使低频也必须保留。

基础款的高频只能触发“稳定器候选”，绝不能自动压过表达力、评分或情感证据更强的单品。

## 6. 搭配与洞察门槛

每套 Best Match 生成构成画像：廓形层级、色彩角色、材质/视觉重量、风格与正式度、设计焦点、场景与身形目标。描述文本的搭配意图必须带原文引用。

洞察只有四个等级：

| 等级       | 门槛                                                                        |
| ---------- | --------------------------------------------------------------------------- |
| 事实       | 来自 confirmed 字段、评分、槽位或可引用原文                                 |
| 初步模式   | 至少两套独立 Best Match；可由 proposed 补充                                 |
| 稳定原则   | 至少三套独立 Best Match，或「明确原文 + 两套结构支持」，并有 confirmed 支撑 |
| 待验证假设 | 样本不足、主要依赖 proposed，或缺少身形/实穿确认                            |

每条洞察必须显示支持的 Best Match 数、confirmed/proposed 覆盖率和证据入口；数据不足时必须直说“证据不足”。

## 7. 输出与操作

审美理解总览固定包含：审美操作系统、搭配语法画布、单品角色网络、个人审美维度剖面、张力与待开发清单，以及证据检查器。它们服务于下一步“试什么、为什么值得试”，不伪装成确定推荐。

身形档案由 Victor 自由填写自然语言，内容可以包含比例感知、修饰目标、避雷项、舒适度和真实穿着验证。不要求身体照片，不录入测量值，不自动推断；分析只能引用这段原话，并把由搭配结构生成的解释标为待验证。

## 8. 版本、重算与变更

- 每个分析包保存输入快照哈希与引擎版本。
- 改变字段解释、权重、阈值、关系规则或洞察模板，必须升级 `engineVersion` 并重算派生包。
- 任何规则变更先更新本文件，再更新实现和验证脚本。
- 测试至少验证：源快照不可变、`proposed=0.55`、关系可溯源、洞察证据完整、高评分未开发单品可被发现。

## 9. 明确禁止

- 用词频、单纯共现或基础款出现次数宣布“审美核心”。
- 让 `proposed` 单独生成正式原则。
- 将评分、频率、兼容度混为一个不可解释分数。
- 从图片推断身形结论。
- 让分析服务在未经确认时回写视觉字段，或触碰 Production。
- 在个人样本不足时引入图数据库、GNN、向量数据库或通用电商穿搭模型来制造“智能感”。
