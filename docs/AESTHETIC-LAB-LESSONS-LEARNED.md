# 衣LOG 二阶段审美实验台：实施复盘与防重犯清单

> 这是二阶段的强制复盘文档。每次继续开发审美基座、图片识别或本地数据同步前，先读 `docs/PROJECT-STATUS.md`、`docs/AESTHETIC-FOUNDATION-EXECUTION-PLAN.md` 和本文。

## 一、已经发生的问题

### 0. 根本问题：没有先完成环境判定与 Development readiness

- **表现**：在 Development 尚未准备好时，直接讨论本地统计、账号同步、OAuth 和 Provider，后续不断用 Preview、Production 只读桥或手动快照补洞。
- **根因**：没有把“环境是否具备工作条件”作为产品方案的第一道闸门；把环境问题误当成局部功能问题处理。
- **今后原则**：先确认 Development 与 Production 的 project ref、迁移、认证、真实账号数据、RLS 和 Provider 状态，再选择技术方案。Development 未准备好时，停止功能开发，不用 Production 绕过去假装完成。
- **固定顺序**：`环境盘点 → Development 补齐 → 真实账号 smoke test → Provider smoke test → 选择本地/线上方案 → 功能开发`。

### 1. 功能计划把 Provider 当成了前置条件

- **表现**：Google OAuth、Supabase、Kimi 服务没有完全准备好时，审美实验仍然被推进，导致一次次卡在“登录/Provider”而不是数据本身。
- **根因**：没有把“本地数据分析核心”“账号数据访问”“图片视觉 Provider”拆成三个独立闸门。
- **改正**：本地统计、关系计算、人工确认必须可离线运行；账号同步从 Production 只读拉取，实验结果留在 localhost；视觉 Provider 失败时降级 RGB 与人工字段，不阻塞统计。

### 2. 本地页面没有登录入口，却提供了账号同步按钮

- **表现**：Victor 已在衣 log 正站登录，本地页面仍提示“请先登录”，页面没有告诉用户应该去哪里登录。
- **根因**：`localhost` 与衣 log 正站是不同 origin，Supabase 会话保存在各自 origin 的浏览器存储中，不能自动共享；本地路由又被刻意放在主应用鉴权闸门之外。
- **改正**：本地页显示 Production 数据源会话状态；没有会话时直接提供数据源登录表单，不再把主应用 Development 登录误当成数据源登录。

### 3. Development 与 Production 的边界没有在用户界面中显式呈现

- **表现**：用户说“我的账号”，系统没有清楚说明当前读取的是哪个 Supabase 项目。
- **根因**：环境隔离只存在于代码和环境变量，缺少用户可见的环境说明与运行前闸门。
- **改正**：实验客户端固定 Development，数据源客户端只允许 `import.meta.env.DEV + VITE_AESTHETIC_SOURCE_ENV=production`；页面同时显示“Production 只读 / localhost 写入”，环境不符时直接拒绝。

### 4. Kimi key 被错误判定为失效

- **表现**：一次自写健康检查得到 401，错误结论是 key 失效；实际线上验证和按正确请求格式的本地测试均为 200。
- **根因**：健康检查没有复用线上已经成功的请求合同，额外发送了 Kimi Code 不接受的参数（`temperature`），没有区分“凭证错误”和“请求合同错误”。
- **改正**：本地中间件与 `api/aesthetic/vision/analyze.ts` 使用相同的 endpoint、model、`max_tokens`、system message 和多模态 messages；Provider 验证必须使用真实视觉请求合同，并记录 HTTP 状态、响应解析状态和字段完整性。

### 5. 数据入口先做了手动快照，违背了“不要重新填”

- **表现**：本地分析台初始只能导入 JSON，真实账号数据还需要用户去另一个页面下载。
- **根因**：先做了分析 UI，再补数据访问，没有把“真实数据进入实验台”列为第一条验收条件。
- **改正**：增加 Production 账号只读同步；实验结果仍只写 localhost，JSON 导入保留为离线备份和故障恢复手段，不再作为唯一入口。

### 6. 根路由登录与数据源登录混在一起

- **表现**：localhost `/` 的登录页使用实验客户端（Development）；Victor 用 Production 账号在这里登录，自然会失败或看不到账号数据。
- **根因**：页面没有把“主应用业务登录”和“审美数据源登录”区分出来。
- **改正**：本地分析台内置独立的 Production 只读登录表单；根路由新增明确的本地分析台入口。以后数据源登录必须在数据源上下文中完成，不能复用默认实验客户端。

### 7. 原生色盘在内置浏览器中点击即收起

- **表现**：`input[type=color]` 的系统弹层打开后立即关闭，无法连续调整颜色。
- **根因**：原生颜色控件的弹层生命周期由内置浏览器接管，和 React 重渲染不稳定。
- **改正**：改为应用内常驻色盘预设，保留 RGB 精确输入；选择颜色后色盘不自动关闭，用户点击“完成”才收起。

### 8. 预设色板不等于自由选色

- **表现**：常驻预设色板虽然不再闪退，但无法覆盖伞兵裤等低饱和、特殊材质的真实颜色。
  - **改正**：改为二维饱和度/明度区域 + 独立色相滑轨的自由 HSB 色盘，RGB 作为精确校正通道；识别结果已有时，按钮变为“重新解析”，允许对当前图片再次请求识别。
  - **补充**：内置浏览器在 pointer capture 后不保证 `event.buttons` 正确，拖拽状态必须由显式 ref 维护，并在 pointer up/cancel/lost capture 时释放；颜色同时保存主色、辅色和点缀色角色。
  - **再次补充**：颜色值变化不能参与 React key，否则第一次拖动就会卸载并重建色盘；颜色行使用稳定索引键，并支持点击组件外任意位置关闭浮层。
  - **当前交互**：颜色字段支持新增与删除，新增上限为 6 个；每条颜色记录仍需人工确认颜色角色。

## 二、今后固定的执行顺序

### 9. 词频与粗共现不能冒充审美分析

- **表现**：早期“关系与结论”把高频基础款、故事中的常见词和单套共现直接放进结论区，输出无法解释搭配为什么成立。
- **根因**：没有区分原始事实、带上下文的关系、评分偏好、文本意图与待验证推断；将计数当成审美理解。
- **改正**：所有新洞察遵循 `docs/AESTHETIC-ANALYSIS-RULES.md`：`proposed` 以 0.55 参与探索，正式原则必须有 confirmed/原文与独立 Best Match 支撑；评分、频率、兼容性严格分开；每条关系与结论必须回溯到单品、槽位、字段或原文。

任何二阶段功能必须按以下顺序通过，不允许跳步：

1. **环境闸门**：明确当前运行环境、Supabase project ref 和数据所有权；Production 只能作为只读数据源，实验写入只允许 localhost。
2. **身份闸门**：页面能显示会话状态；无会话时能在当前 origin 完成登录并自动返回。
3. **数据闸门**：先用 Production 只读真实账号数据完成同步，记录单品数、Best Match 数和失败原因；再开始分析功能。
4. **Provider 闸门**：用线上成功合同做文本和图片 smoke test；区分缺 key、401、请求参数错误、超时、返回不可解析 JSON。
5. **离线降级**：Provider 不可用时，统计、关系、人工录入和 RGB 像素提取仍可运行。
6. **功能验收**：只有前五项都通过，才进入新的关系推导或 UI 交互开发。

### Development readiness 必须全部通过

- Supabase project ref 与 Vercel Preview 的 `VITE_SUPABASE_URL` 一致，且明确标记 `development`；
- 所有二阶段 migration 已在 Development 应用，并有 dry-run/回滚记录；
- Development 有 Victor 可用的登录方式，能读取自己的单品和 Best Match；
- RLS 允许本人读取、拒绝其他用户读取；
- Kimi endpoint、model、key 在 Development 本地服务中用真实文本和图片合同返回 200；
- 本地分析台默认只连接 Development，页面明确显示环境；
- 完成以上检查后，才决定是否需要 Production 只读导入或其它桥接方案。

## 三、固定验收清单

- 页面是否显示“Production 数据源只读 / localhost 分析写入”状态？
- 未登录时是否有明确登录入口，并能返回原页面？
- 是否证明读取的是 Production 只读数据源，且没有任何写入？
- 是否实际读到账号单品和 Best Match，而不是示例或空快照？
- Provider 是否用真实图片请求合同返回 200？
- Provider 失败时是否保留统计和人工流程？
- 是否记录了原始错误码、根因和降级路径？
- 是否运行 `npm run lint` 和 `npm run build`？
- 是否更新本文件和执行计划，而不是只在聊天中解释？

## 四、当前调整后的产品路线

当前阶段只做三件事：

1. 本地 Production 账号只读同步与快照持久化；
2. 本地统计/关系/文本线索分析台；
3. 图片识别字段的 Provider 识别、RGB 降级和 Victor 确认闭环。

快速搭配、实验台完整交互、好友邀请和跨用户能力全部等待这三件事通过真实数据验收后再排期。
