## 历史原始检查（迁移前，不作为当前上线清单）

以下安全、成本和用户体验条目记录于 Firebase/Firestore 时代。当前整改状态以本文件“当前复核结论”、`docs/PROJECT-STATUS.md` 和 `docs/DEPLOYMENT-ENV.md` 为准。

## 一、安全 / 成本（历史）

### 🔴 P0 必须 — `/api/ai-import` 完全无鉴权、无限流、无大小限制
- 文件：`api/ai-import.ts`。任何人都能 `POST /api/ai-import` 直接消耗服务端 `KIMI_API_KEY`（真金白银 + 可能打爆 Kimi 额度，连累正常导入）。链接一旦半公开，这就是一个开放的「烧钱水龙头」。
- 建议：① 要求合法 Firebase ID Token（轻量做法：用 `jose` 对 Google securetoken 公钥验签 + 校验 aud/exp，**不引入 firebase-admin**，与 `api/public` 既有的免 SDK / REST 风格一致）；② 限制请求体大小；③ 基础限流（按 uid / IP）。最低限度先做「必须登录」即可挡掉 ~90% 的匿名滥用。

### 🟡 P1 应做 — 写入侧无滥用上限（存储 / 额度）
- `firestore.rules` 有意不校验字段大小（为了存 base64）。已登录用户可写超大文档 → 撑爆 1GiB 存储 / 抬高读写额度。客户端虽在 `AddEditItemModal` 限图 5MB，但可绕过。
- 建议：规则里加文档/图片字段长度上限（如 base64 串 ≤ ~1.5MB），可选加「每用户单品数上限」。半公开下值得做。

### 🟡 P1 应做 — `/migrate` 开发工具对所有登录用户开放
- `App.tsx` 路由 + `MigrateData.tsx`。非破坏性（只 `addDoc` 拷贝，且跨用户拷贝只对**已开分享**的柜生效），但属内部一次性工具，不该暴露给试用者。
- 建议：删掉该路由，或仅对作者 UID 放行。低成本。

### 🟢 P2 建议 — 清理 `.env` 里遗留的 `VITE_KIMI_API_KEY`
- 已核验：**当前打包产物未泄露该值**（代码无任何 `import.meta.env.VITE_KIMI` 引用，未被 inline）。但 `VITE_` 前缀的密钥是隐患——日后一旦有人引用就会进客户端 bundle。
- 建议：从本地 `.env` 与 Vercel 环境变量删除它。真正在用的是服务端 `KIMI_API_KEY`（无前缀，`api/ai-import.ts` 用）。

### 🟢 现状良好（无需改）
- 数据按 `userId` 隔离，规则正确（本人读写 / 分享只读）。
- 公开读全部走 `api/public/[uid].ts` 边缘缓存 + 显式 `shareEnabled` 闸门；`api/img` 校验 ownerId、仅服务已分享的柜。
- `wardrobe_users` 世界可读但只暴露 `shareEnabled` 布尔，无 PII，属设计需要。
- Google 专属登录（已移除匿名），WebView 检测到位。

### 成本 / 容量提醒（非漏洞，但影响体验稳定性）
- Firestore 免费层 **5万读/天硬上限**（此命名库开账单也提不了，见 `容量评估.md`）。刚上线的本地持久缓存 + 边缘缓存已大幅缓解；单开容量约 ~350 日活。
- 小红书 burst 下，公开卡墙/作者预览是边缘缓存（与访问量解耦，压力小）；真正吃额度的是**新注册者各自加载 + 作者自己的重 base64 读**。试用规模大概率够用；若起量会看到「服务器繁忙」面板（已优雅处理）。彻底解需迁标准库（`容量评估.md`）。

## 二、用户体验

### 🟡 P1 — 微信/WebView 内打不开 Google 登录，且没有「复制链接」按钮（小红书人群最大摩擦）
- `App.tsx` `LoginPage` 的 `isWebView` 分支只给了「请在系统浏览器打开」的**纯文字提示**，没有一键复制链接的按钮。小红书用户常在 App 内置浏览器点链接 → 无法登录 → 大概率流失。
- 建议：WebView 分支加「一键复制链接」按钮 + 更清楚的「在 Safari/Chrome 打开」引导。**这是小红书渠道转化率的关键一环。**

### 🟡 P1 — 音效无静音开关
- `sounds.ts`：每次 hover 卡片都播 Thump，无全局静音。音量低（0.09–0.35）且需用户交互后才响，但满屏卡片 hover 会持续出声，部分人会觉得吵。
- 建议：加静音开关（存 localStorage），或去掉 hover 音。

### 🟢 P2 — 新用户首屏混入「power-user / 内部」按钮
- `WardrobeList` 工具栏对**空衣柜新用户**也显示「导入数据 / 清理重复 / 分享」。「清理重复」尤其像内部数据维护功能，易困惑。
- 建议：空衣柜时弱化/隐藏「清理重复」「导入数据」，让首次体验聚焦「添加衣物」。（空态示例卡已优化得不错。）

### 🟢 P2 — 缺反馈入口（与「看反响」目标直接相关）
- 没有任何反馈/联系入口。既然目标是「看反响」，建议加一个轻量反馈链接（小红书私信 / 表单 / 邮箱），ROI 高。

### 🟢 P2 — 移动端真机冒烟
- 布局是响应式的，但建议真机过一遍：BestMatch 拼贴 builder、TagBundle、以及 iOS Safari / Android Chrome 的登录往返。（此环境无法代验。）

### 🟢 现状良好
- 读失败已区分「繁忙重试」vs「空衣柜」，示例卡已标注「来自作者的衣柜·这不是你的衣物」（本轮前序改动已上线）。
